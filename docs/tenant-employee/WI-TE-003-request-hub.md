# WI-TE-003 휴가 및 요청 허브 상세 설계

## 작업 ID

- `WI-TE-003`

## 변경 배경

기존 `TE-201`은 범용 요청 허브로 정리되어 있었다.
하지만 업종별 시나리오를 다시 보면,
오피스형 직원과 리테일형 직원은 요청 허브에서 가장 자주 여는 항목이 다르다.

따라서 `WI-TE-003`은 이제 범용 요청 허브가 아니라
`업종 팩별 요청 허브 변형`을 설계하는 작업으로 재정의한다.

## 현재 설계 범위

### Variant A. Office Pack Request Hub

대상:

- 사무직 중심 조직
- 휴가 / 일반 품의 / 문서 / 정정 요청 사용 빈도가 높은 조직

핵심 목적:

- 자주 쓰는 요청을 빠르게 시작
- 정책 요약 확인
- 반려 / 재제출 관리
- 내 요청 상태 추적

핵심 결정:

- 오늘 어떤 요청을 제출해야 하는가
- 어떤 요청이 반려되어 재제출이 필요한가
- 정책상 주의할 점이 무엇인가

산출물:

- [request-hub-office.html](../../wireframes/tenant-employee/request-hub-office.html)

### Variant B. Retail Pack Request Hub

대상:

- 매장형 / 교대형 현장 직원

핵심 목적:

- 출퇴근 정정
- 시프트 변경 응답
- 대체 근무 회신
- 초과근무 / 결원 대응 요청

핵심 결정:

- 오늘 근무 관련 예외를 어떻게 보고할 것인가
- 변경된 시프트에 응답해야 하는가
- 출퇴근 정정과 대체 근무 요청 중 무엇이 우선인가

산출물:

- [request-hub-retail.html](../../wireframes/tenant-employee/request-hub-retail.html)

## 공통 설계 원칙

- 요청 허브는 메뉴 모음이 아니라 `행동 시작점`이어야 한다
- 요청 유형은 업종 팩에 따라 우선순위가 달라져야 한다
- 정책 힌트는 짧고 행동 직전 문맥에 있어야 한다
- 최근 요청과 반려 재제출 흐름이 바로 이어져야 한다

## Office Pack 초점

- 휴가 요청
- 일반 품의
- 출퇴근 정정
- 문서 관련 요청
- 반려 / 재제출

## Retail Pack 초점

- 출퇴근 정정
- 시프트 변경 수락 / 거절
- 대체 근무 응답
- 초과근무 요청
- 매장 지원 요청
