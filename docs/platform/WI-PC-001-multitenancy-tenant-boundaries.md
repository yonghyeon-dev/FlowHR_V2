# WI-PC-001 멀티테넌시 아키텍처와 테넌트 경계 정의

## 작업 ID

- `WI-PC-001`

## 목적

FlowHR_V2는 하나의 HR SaaS 안에서 여러 고객사를 안전하게 운영해야 한다.
따라서 멀티테넌시는 단순 데이터 분리 개념이 아니라, 인증, 권한, 로그, 파일,
리포트, 기능 플래그, 운영 툴까지 모두 포함하는 제품 경계로 정의해야 한다.

## 핵심 원칙

- 모든 업무 데이터는 반드시 `tenant_id`를 기준으로 분리한다.
- 테넌트 경계는 UI, API, 배치, 파일 스토리지, 검색 인덱스, 로그 조회 권한에 동일하게 적용한다.
- `Platform Console`은 테넌트 운영 메타 정보에 접근할 수 있지만, 고객사 업무 데이터 원문 접근은 별도 통제를 둔다.
- 같은 사용자가 여러 테넌트에 속할 수 있어도 세션 컨텍스트는 항상 단일 테넌트 기준으로 고정한다.
- 테넌트 전환은 명시적 액션이어야 하며, 무의식적 컨텍스트 혼합이 일어나면 안 된다.

## 경계가 필요한 자산

### 1. 업무 데이터

- 직원 프로필
- 근태 기록
- 휴가 정책 및 사용 이력
- 결재 문서
- 급여 정산 데이터
- 리포트 결과

### 2. 운영 메타 데이터

- 테넌트 플랜
- 계약 상태
- 좌석 수
- 도메인 / SSO 설정
- 기능 플래그 상태
- 지원 티켓 메타 정보

### 3. 보안 자산

- 로그인 이력
- 관리자 활동 로그
- 민감 액션 승인 기록
- 파일 접근 로그

## 테넌트 경계 규칙

### UI 계층

- 고객사 관리자와 직원은 자기 테넌트 데이터만 본다.
- URL, 검색 결과, 최근 항목, 인박스는 현재 테넌트 컨텍스트만 반영한다.
- 테넌트명이 같은 다른 조직과 혼동될 수 있으므로, `Tenant Admin` UI에는 회사 브랜딩 고정 영역이 필요하다.

### API 계층

- 모든 요청은 인증 주체와 `tenant_id`를 함께 검증한다.
- `tenant_id`는 클라이언트 입력만 신뢰하지 않고 서버 세션/토큰과 교차 검증한다.
- 교차 테넌트 접근 시도는 실패 응답과 함께 감사 로그를 남긴다.

### 검색 / 리포트 계층

- 검색 인덱스는 논리적으로 tenant scoped query를 강제한다.
- 예약 리포트와 내보내기 파일은 생성 시점의 테넌트 컨텍스트를 보존한다.
- 다운로드 URL은 재사용 가능한 공개 링크가 아니라 짧은 만료 정책을 기본으로 한다.

### 파일 / 문서 계층

- 파일 저장 경로는 tenant partition 구조를 따른다.
- 문서 원본과 서명본은 감사 가능해야 하며 tenant boundary가 항상 명시되어야 한다.

## 운영자 예외 접근 모델

- `Platform Operator`는 기본적으로 테넌트 메타 정보만 조회한다.
- 고객사 업무 데이터 원문 열람은 `break-glass` 또는 승인형 접근으로 제한한다.
- 원문 열람 시 목적, 승인자, 시간, 대상 테넌트를 남겨야 한다.

## 세션 컨텍스트 규칙

- 한 세션에서 동시에 2개 이상 테넌트 업무 화면을 처리하지 않는다.
- 테넌트 전환 시 전역 검색, 최근 항목, 임시 draft, 열려 있는 패널을 초기화한다.
- 위험한 작업 전에는 현재 테넌트 배지를 반복 노출한다.

## 설계 시 체크리스트

- 이 화면이 현재 테넌트 컨텍스트를 명확히 보여주는가
- 최근 항목 / 검색 결과가 다른 테넌트 데이터를 섞지 않는가
- 파일, 리포트, 인쇄물에 tenant marker가 남는가
- 운영자 예외 접근이 일반 권한과 구분되는가
