# FlowHR_V2 단계 게이트

## 목적

설계 진행 상태를 감으로 해석하지 않고, 문서 기준으로만 판정하기 위한 기준 문서다.

## 운영 원칙

- 단계 판정은 이 문서 기준으로만 한다.
- 문서 수가 늘었다고 상위 단계를 완료로 올리지 않는다.
- 업종 팩 구조가 필요한 화면은 `Pack 번들 승인`까지 있어야 다음 단계 진입 근거가 된다.
- 이후 보고는 항상 `현재 단계`, `충족`, `미충족`, `다음 작업` 순서로 정리한다.

## Stage 1. 방향 정의

### 완료 기준

- 제품 방향 문서가 있다
- 권한 레이어가 정의돼 있다
- 경쟁 제품 참고 방향이 정리돼 있다

### 현재 판정

- 완료

## Stage 2. 권한별 화면 초안

### 완료 기준

- Platform / Tenant Admin / Tenant Employee 축이 분리돼 있다
- 역할별 주요 화면 와이어 초안이 있다
- 화면 맵 초안이 있다

### 현재 판정

- 완료

## Stage 3. IA 계층 설계

### 완료 기준

- 대분류, 중분류, 소분류, 상세분류가 문서화돼 있다
- 각 분류가 화면 ID와 연결돼 있다
- 권한별 메뉴/기능/행동 매핑이 있다

### 현재 판정

- 완료

## Stage 4. 화면 상세 설계

### 완료 기준

- 핵심 화면 와이어와 WI 문서가 존재한다
- 화면 목적, 핵심 결정, 상태, 액션이 문서화돼 있다
- 업종 차이가 필요한 화면은 Pack 변형이 분리돼 있다
- 코어 화면 승인 로그가 있다
- Bundle 단위 리뷰 패킷이 존재한다

### 현재 판정

- 완료

- 공통 컴포넌트 규격서가 있다

- 완료

### 완료 기준

### 완료 기준

- 필요한 Pack 번들 승인 로그가 `approved` 상태다

### 현재 판정

- 미도달

## 현재 공식 상태

- 현재 단계: `Stage 4 진행 중, Stage 6 완료`
- 정확한 표현: `상세 설계 진행 단계`
- 금지 표현:
  - `구현 직전`
  - `설계 완료`
  - `바로 구현 가능`

## 현재 판정 근거

### Stage 4

- 코어 화면 와이어와 WI 문서는 존재한다
- Office / Retail Pack 변형과 Platform Risk-first 변형도 분리됐다
- 번들 승인 로그는 생성됐지만 아직 모두 `hold`다

### Stage 6

- API, 라우트, i18n, 컴포넌트 연결 문서가 정리돼 있다
- 구현 핸드오프 패키지와 승인-구현 브리지 문서도 존재한다

### Stage 7 미도달 사유

- Office / Retail Bundle 승인 로그가 아직 `hold`
- Screen 승인 로그도 일부만 승인 상태다
- 따라서 구현 직전 단계로 올릴 수 없다
