# FlowHR_V2 제품 방향 v1

## 목적

FlowHR_V2는 `Shiftee`와 `flex`를 벤치마크로 삼되, 단순 기능 복제가 아니라
운영자가 매일 오래 쓰는 HR SaaS의 밀도와 사용성을 목표로 한다.

핵심 지향점은 다음과 같다.

- 운영형 HR SaaS: 숫자 조회보다 승인, 예외, 병목 처리 흐름을 우선한다.
- 권한 분리형 구조: SaaS 운영 권한과 고객사 운영 권한을 명확히 분리한다.
- 프로덕션 기준 범위: MVP 축소가 아니라 실제 상용 서비스 기준으로 설계한다.
- UI/UX 우선 접근: PRD를 확정하기 전에 화면 구조와 정보 우선순위를 먼저 합의한다.

## 권한 레이어

### 1. Platform Operator

우리 SaaS를 운영하는 내부 권한이다.

- 테넌트 생성/정지/복구
- 플랜, 과금, 청구 운영
- 지원, 공지, 장애 대응
- 보안 정책, 감사 로그 운영

### 2. Tenant Admin

고객사 관리자 권한이다.

- 조직/직원 정보 운영
- 근태, 휴가, 문서, 승인, 급여 운영
- 리포트와 정책 관리
- 역할 및 권한 위임

### 3. Tenant Employee

고객사 직원 셀프서비스 권한이다.

- 출퇴근, 일정, 휴가 요청
- 문서 서명
- 요청 상태 확인
- 개인 정보와 잔여 현황 조회

`Tenant Manager`, `Payroll Admin`, `Recruiter`는 독립 최상위 권한이 아니라
`Tenant Admin` 하위 역할로 정의한다.

## 제품 면 분리

### Platform Console

SaaS 운영 콘솔이다. 고객사 업무 화면과 분리해야 한다.

### Tenant HR App

고객사가 실제로 사용하는 HR 운영 제품이다.

- 관리자용 웹 앱
- 직원용 웹/모바일 셀프서비스

## UX 방향

- 홈은 요약 화면이 아니라 운영 큐의 시작점이어야 한다.
- 예외와 승인 병목은 KPI보다 먼저 읽혀야 한다.
- 직원 화면은 메뉴 탐색보다 즉시 행동을 우선해야 한다.
- 관리자 화면은 표 중심이라도 지루하면 안 된다. 밀도와 가독성을 함께 잡아야 한다.

## 설계 프로세스

1. 경쟁 서비스 참고
2. 핵심 화면 와이어프레임
3. 기본 UX 컨펌
4. 대분류 > 중분류 > 소분류 > 상세분류 설계
5. `WI-*` 작업 항목 분해
6. 이후 구현
