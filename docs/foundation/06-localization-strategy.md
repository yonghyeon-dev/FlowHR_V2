# FlowHR_V2 로컬라이제이션 전략

## 목적

FlowHR_V2는 브라우저 언어, 고객사 기본 언어, 사용자 명시 언어 설정을 기준으로
한국어와 영어를 동적으로 전환할 수 있어야 한다. 이 문서는 텍스트 키 설계와
런타임 언어 결정 규칙의 기준을 정의한다.

## 언어 결정 우선순위

1. 사용자 명시 언어 설정
2. 고객사 기본 언어
3. 브라우저 언어
4. 최종 fallback `en`

와이어프레임 검토 단계에서는 `?lang=ko`, `?lang=en` 쿼리 파라미터를 우선 적용한다.

## 키 설계 원칙

### 기본 형식

- `LID-{ROLE}-{MODULE}-{SEMANTIC_KEY}`

예시:

- `LID-TE-HOME-TODAY_STATUS_TITLE`
- `LID-TE-INBOX-SIGN_REQUIRED_BADGE`
- `LID-TA-PERFORMANCE-EVAL_CYCLE_TITLE`
- `LID-DM-LEAVE-HALF_DAY_POLICY_HINT`

### 규칙

- 키는 화면 위치가 아니라 의미를 표현한다.
- IA가 바뀌어도 최대한 키를 유지할 수 있게 만든다.
- 숫자나 상태값은 문자열에 하드코딩하지 않고 변수로 치환한다.
- 메뉴명, 상태 배지, 경고 문구, 빈 상태 문구도 별도 키로 관리한다.

## 변수 치환 규칙

- 기본 문법은 `{name}` 형식을 사용한다.
- 예시:
  - `ko`: `처리할 항목 {count}건`
  - `en`: `{count} pending items`

날짜, 시간, 숫자 표시는 런타임에서 locale 기준 포맷팅한다.

## 리소스 구조

권장 구조:

```json
{
  "LID-TE-HOME-TODAY_STATUS_TITLE": {
    "ko": "오늘 상태",
    "en": "Today status"
  }
}
```

## 런타임 동작 원칙

- 페이지 로드 시 현재 locale을 결정한다.
- `data-lid`가 있는 요소에만 번역을 적용한다.
- locale 값이 없으면 `en`, 그 다음 첫 번째 사용 가능한 값을 fallback으로 사용한다.
- 필요 시 `localStorage`에 사용자 선택 언어를 저장한다.

## 설계 적용 범위

- 직원용 모바일/웹 셀프서비스 화면
- 관리자용 운영 화면
- 상태 배지, 인박스, 알림, 문서 처리 문구
- 이후 실제 앱 구현 시 API 에러 메시지와 빈 상태 문구까지 확장

## 주의 사항

- `LID-대분류-중분류-소분류-상세분류`처럼 구조를 그대로 키에 넣지 않는다.
- 번역 키를 컴포넌트 파일명이나 DOM 구조에 종속시키지 않는다.
- 단순 번역이 아니라 역할별 문맥 차이를 고려한다.

## 현재 적용 상태

- 공유 i18n 스크립트 추가
- 신규 와이어프레임부터 `LID-*` 키 적용
- 기존 초기 와이어는 순차적으로 마이그레이션 예정
