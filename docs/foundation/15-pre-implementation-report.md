# FlowHR_V2 구현 전 종합 보고서

## 현재 단계

FlowHR_V2는 `기획 초안`이 아니라 `구현 직전 설계 패키지` 단계까지 확장되었다.
권한 모델, 화면 체계, 주요 와이어, 공통 도메인 규칙, 데이터 계약 초안,
다국어 구조, 컴포넌트 인벤토리까지 이어진 상태다.

## 이번 패스에서 마감한 항목

- Platform 와이어 4종의 잔여 정적 텍스트를 `LID-*` 기준으로 정리
- 구현용 `src/i18n/locales` 시드 카탈로그 구조 생성
- 공통 컴포넌트 인벤토리 정의
- 화면 준비도 매트릭스 정리

## 구현 착수 기준에서 확보된 것

- `Platform / Admin / Employee / Auth` 라우트 축이 있다
- 주요 화면이 권한 레이어별로 분리되어 있다
- 핵심 화면 와이어가 승인 검토 가능한 상태다
- 우선 구현 화면의 API 계약 초안이 있다
- `LID-*` 전략과 리소스 파일 구조가 있다
- 공통 상태, 예외, 권한 거부, step-up UX 기준이 있다

## 아직 구현 시점에 확정해야 하는 것

- 실제 프레임워크 기준 라우팅 방식
- OpenAPI 또는 RPC 스키마의 최종 형식
- 상태관리 라이브러리 선택
- 디자인 토큰과 정식 프로덕션 UI 스타일
- 인증 공급자와 세션 인프라 구체안

## 구현 권장 착수 순서

1. `src/app` 라우트 셸과 공통 i18n 로더 구성
2. `Tenant Employee` 홈, 출퇴근, 요청 허브
3. `Tenant Admin` 홈, People, Attendance, Leave
4. `Platform` 콘솔과 보안/과금 화면
5. Payroll, Recruiting, Reports 확장 모듈

## 위험과 주의점

- 현재 와이어는 승인용 저해상도 화면이므로 정식 UI 스타일과 동일시하면 안 된다.
- `R3` 화면은 구현 전에 모듈 단위 API와 번역 리소스 매핑을 더 구체화해야 한다.
- 권한/테넌시 경계는 UI가 아니라 서버 계약에서 최종 강제되어야 한다.

## 결론

요청한 방향대로 설계를 계속 확장한 결과, 지금 상태는
`구현 바로 직전의 종합 설계 단계`로 보는 것이 맞다.
다음 단계부터는 새 설계 영역을 더 넓히기보다,
정해진 기준 위에서 실제 앱 골격과 화면 구현으로 넘어가는 것이 자연스럽다.
