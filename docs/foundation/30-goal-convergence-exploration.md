# FlowHR_V2 목표 수렴 탐색 프레임

## 목적

이 문서는 현재 설계가 왜 흔들렸는지 구조적으로 정리하고,
앞으로 어떤 탐색 순서로 목표를 수렴해야 하는지 고정하기 위한 기준 문서다.

핵심 목적은 세 가지다.

- `범용 HR SaaS` 수준의 추상 설계에서 벗어나기
- `업종별 운영 맥락`을 먼저 고정한 뒤 화면을 다시 설계하기
- 문서 수나 와이어 수가 아니라 `의사결정 품질`로 승인 여부를 판단하기

## 현재 관찰된 문제

### 1. 벤치마크는 있었지만 목표 고객 정의가 약했다

- `Shiftee`, `flex`를 벤치마크로 삼았지만
- 어떤 업종, 어떤 조직 규모, 어떤 운영 복잡도를 먼저 잡을지 명확하지 않았다
- 그 결과 와이어가 `누구에게나 조금 맞는 화면`으로 흐르기 쉬웠다

### 2. 권한 구조는 있었지만 운영 맥락 축이 비어 있었다

- `Platform Operator / Tenant Admin / Tenant Employee` 분리는 맞다
- 하지만 같은 `Tenant Admin`이라도 업종에 따라 첫 화면과 우선순위가 크게 다르다
- 이 축이 빠진 상태에서 화면을 먼저 만들다 보니 공통 구조만 남고 맥락이 빠졌다

### 3. 화면 승인 패킷이 너무 화면 중심이었다

- 현재 승인 패킷은 `어떤 화면을 볼지`는 정리돼 있다
- 반대로 `이 화면이 어느 업종의 어느 결정 순간을 해결하는지`는 약하다
- 그래서 승인 판단이 시각 배치나 박스 구성 수준으로 흐를 위험이 있었다

### 4. 실패 기준보다 산출물 기준이 먼저였다

- 화면이 존재한다
- 문서가 연결돼 있다
- `LID-*`가 붙어 있다

이 세 가지는 진행 증거일 뿐, 제품 적합성 증거는 아니다.
앞으로는 `왜 이 화면이 실무에서 버려질지`를 먼저 검토해야 한다.

## 목표 수렴의 기준 축

앞으로 목표 수렴은 아래 5개 축을 먼저 고정한 뒤에만 진행한다.

### 1. 우선 공략 업종

최소 아래 중 `1순위 2개`, `후순위 1개`를 확정해야 한다.

- 오피스형 조직
- 리테일/매장형 조직
- 제조/현장형 조직
- 물류/교대형 조직
- 케어/전문자격형 조직

### 2. 조직 운영 형태

- 단일 사업장
- 다지점 운영
- 현장/본사 혼합
- 고정 근무 중심
- 유연근무 중심
- 교대 근무 중심

### 3. 관리자 의사결정 유형

각 업종마다 관리자가 매일 하는 결정이 무엇인지 먼저 정의해야 한다.

예시:

- 오늘 결원 인력을 어디에 재배치할 것인가
- 오늘 승인 적체 중 무엇을 먼저 처리할 것인가
- 이번 마감에서 급여 오류를 유발할 예외는 무엇인가
- 어떤 계약/서명이 법적 리스크를 만들고 있는가

### 4. 직원 셀프서비스 우선 행동

직원이 이 제품을 여는 이유가 무엇인지 먼저 정해야 한다.

- 출퇴근
- 휴가 요청
- 정정 요청
- 서명
- 일정 확인
- 급여/잔액 확인

### 5. 비목표 범위

무엇을 당장 잘하지 않을지도 명시해야 한다.

예시:

- 1차 공략 업종이 아닌 화면 우선순위
- 고급 분석 기능
- 산업별 특수 규정 자동화
- 커스텀 보고서 빌더

## 권장 탐색 순서

### Step 1. ICP와 비ICP를 먼저 고정한다

현재 가장 먼저 필요한 문서는 `누구를 위한 제품인지`보다
`누구를 지금 우선하지 않을지`까지 포함한 고객 정의다.

산출물:

- 우선 공략 업종 2개
- 후순위 업종 1개
- 제외 업종
- 조직 규모 가정
- 운영 복잡도 가정

### Step 2. 업종 팩과 기능 선택 구조를 정의한다

업종별로 기본 모듈과 기본 정책, 기본 홈 구성이 달라져야 한다.
따라서 `공통 코어 + 업종 팩 + 기능 팩` 구조로 정의하는 편이 맞다.

권장 구조:

- 공통 코어
  - 조직/직원
  - 권한
  - 문서
  - 승인
  - 감사
- 업종 팩
  - 오피스형
  - 리테일/매장형
  - 제조/현장형
  - 물류/교대형
- 기능 팩
  - 근태
  - 시프트
  - 휴가
  - 전자문서
  - 급여
  - 성과
  - 채용

이 단계에서 결정해야 할 것:

- 업종별 기본 홈 우선순위
- 업종별 기본 KPI
- 업종별 위험 신호
- 업종별 기본 용어
- 업종별 기본 메뉴 순서

### Step 3. 권한이 아니라 결정 순간 기준으로 다시 맵핑한다

현재 문서는 권한과 화면 기준으로는 많이 정리돼 있다.
이제는 `화면 -> 역할`이 아니라 `결정 -> 역할 -> 화면` 순서로 다시 맵핑해야 한다.

예시 형식:

- 결정: 오늘 법규 리스크가 큰 근태 예외를 먼저 처리한다
- 역할: Tenant Admin
- 업종: 제조/현장형
- 필요한 정보: 연속근무, 휴게 위반, 교대 인수인계 공백
- 필요한 액션: 승인, 재배치, 정정 요청, 경고 발송
- 연결 화면: Attendance Dashboard, Exceptions, Shift Board

### Step 4. 홈 화면을 업종별로 다시 설계한다

홈은 공통 와이어 하나로 버티기 어렵다.
최소한 아래 둘은 분리해야 한다.

- 공통 운영 허브
- 업종별 우선순위가 반영된 홈 변형

예시:

- 오피스형 관리자 홈
  - 승인, 문서, 유연근무, 성과
- 리테일/매장형 관리자 홈
  - 결원, 피크타임 커버, 매장별 스케줄
- 제조/현장형 관리자 홈
  - 교대 커버리지, 휴게 위반, 야간 연속근무

### Step 5. 비관적 검토 루프를 통과한 화면만 승인 후보로 올린다

앞으로 승인 후보는 아래 질문을 통과해야 한다.

- 이 화면이 어떤 업종에서 바로 깨지는가
- 라벨 없는 시각 장식이 있는가
- 숫자는 있는데 액션이 없는가
- 이 사용자가 오늘 여기서 무엇을 결정하는가
- 같은 목적을 더 적은 구역으로 더 명확하게 표현할 수 있는가

## 추천 산출물

목표 수렴을 위해 다음 문서 세트가 필요하다.

### 1. 업종 세분화 문서

- 목표: 우선 공략 업종과 제외 업종 확정
- 예상 파일: `31-industry-segmentation.md`

### 2. 업종 팩 / 기능 선택 문서

- 목표: 공통 코어, 업종 팩, 기능 팩 정의
- 예상 파일: `32-industry-pack-feature-selection.md`

### 3. 역할별 의사결정 시나리오 맵

- 목표: 역할이 아니라 결정 순간 기준으로 화면 재정렬
- 예상 파일: `33-role-decision-scenario-map.md`

### 4. 승인 리셋 계획

- 목표: 기존 `hold` 승인 후보를 어떤 기준으로 재검토할지 정의
- 예상 파일: `34-approval-reset-plan.md`

## 현재 문서에 대한 해석

기존 문서들은 폐기 대상은 아니다.
다만 아래처럼 재분류하는 편이 맞다.

- 유지:
  - 권한 구조
  - IA 계층표
  - 디자인 토큰
  - 공통 컴포넌트 규격
  - 로컬라이제이션 전략
- 재검토:
  - `PC-001`, `TA-001`, `TE-001` 계열 홈 화면
  - 승인 리뷰 패킷의 1차 구현 진입 후보 세트
- 보조 자료화:
  - 이미 만든 범용 와이어프레임 다수

즉 기존 산출물은 `정답`이 아니라 `초기 탐색 흔적`으로 취급하는 편이 맞다.

## 지금 시점의 권장 결론

지금 당장 새로운 화면을 더 그리는 것보다 우선순위는 아래가 맞다.

1. 우선 공략 업종을 좁힌다
2. 업종 팩과 기능 선택 구조를 정의한다
3. 역할별 의사결정 시나리오를 정리한다
4. 그 기준으로 승인 후보 세트를 다시 구성한다
5. 그 다음 홈 화면부터 다시 설계한다
